# AI原生MD编辑器 — 完整产品方案

## 一句话定位

**写了就是知识，不用排版，不用管理，越写越聪明。**

让不懂markdown的普通人，用纯文本写作，AI自动加格式、建链接、归类，自动积累成LLM Wiki知识库。

---

## 核心问题

| 人群 | 痛点 | 现有方案的问题 |
|------|------|---------------|
| 学生/研究员 | 想法很多但不成体系 | Obsidian要学md语法，Notion要手动组织 |
| 产品经理/运营 | 写完文档就过时 | 没人维护交叉引用和一致性 |
| 作家/博主 | 查资料和写作割裂 | 开10个tab查资料，切回来写，来回切换 |
| 任何普通人 | 不会markdown | 被迫用Word或备忘录，知识不积累 |

**核心矛盾：知识管理的价值在"结构和连接"，但维护结构的成本太高，普通人坚持不了。**

---

## 产品哲学

```
1. 用户不需要懂markdown — AI负责结构化
2. 用户不需要手动管理 — AI负责链接、归类、维护
3. 用户不需要记得写过什么 — AI负责检索和关联
4. 越用越聪明 — 每写一段都在给知识库添砖加瓦
```

**类比：**
- Notion让普通人不用写代码就能做数据库
- 我们让普通人不用写markdown就能拥有知识库

## 完整功能蓝图

### 第一层：写作（核心）

#### 1. 零语法纯文本写作
- 用户打开编辑器，直接打字
- 没有markdown工具栏，没有源码/预览切换
- 所见即所得，像Word/Notion一样自然
- AI实时把纯文本变成结构化markdown（后台进行，用户无感）

#### 2. 自动结构化（三层流水线）

```
Layer 1 — 实时（打字时，流式）：
  → 句号后自动分段
  → 列表自动编号
  → 关键词自动加粗
  → 用轻量模型（DeepSeek/GPT-4o-mini）

Layer 2 — 段落级（回车后，debounce 2秒）：
  → 识别段落主题，加合适的heading
  → 识别实体名称，自动加[[]]链接
  → 工具调用：search_notes() 搜索已有笔记
  → 工具调用：web_search() 补充外部知识（可选）

Layer 3 — 文档级（保存时，后台异步）：
  → 更新index.md（目录）
  → 更新log.md（操作日志）
  → 补充交叉引用（其他笔记链接回新内容）
  → 检测与已有内容的矛盾
  → 用重模型（Claude/DeepSeek）
```

#### 3. 语音写作（差异化）
- 用户说话 → AI直接生成结构化markdown
- 不是简单语音转文字，是实时结构化：
  - "今天开会讨论了三个方案" → 自动生成三级标题+列表
  - "方案一的优点是成本低" → 自动识别为对比项

### 第二层：知识库（自动积累）

#### 4. LLM Wiki三层结构（自动维护）

```
notes/
├── index.md          ← AI自动更新的目录
├── log.md            ← AI自动追加的操作日志
├── raw/              ← 用户拖入的原始素材（PDF/网页/截图）
│   ├── articles/
│   ├── papers/
│   └── assets/
├── notes/            ← 用户的笔记（自动结构化后的）
│   ├── concepts/     ← AI自动识别的概念页
│   ├── entities/     ← AI自动识别的实体页
│   └── daily/        ← 日志类笔记
└── SCHEMA.md         ← 自动生成的标签分类和规范
```

**用户完全不需要知道这个结构存在。** AI在后台维护一切。

#### 5. 自动链接和交叉引用

用户写：
```
"今天读了Karpathy的文章，讲LLM自动维护知识库"
```

AI自动：
```
"今天读了[[Karpathy]]的文章，讲[[LLM Wiki]]自动维护知识库。
这跟之前写的[[知识管理演进]]是同一个方向。"
```

#### 6. 拖拽Ingest

```
拖入PDF → AI自动：
  1. 提取全文
  2. 生成摘要笔记
  3. 提取关键概念，创建/更新概念页
  4. 建立交叉引用
  5. 更新index.md和log.md
  6. 弹出通知："新增3个概念页，更新2个已有页面"
```

#### 7. 向知识库提问

```
用户："我之前写过关于知识管理的内容吗？"
  → AI搜索notes/目录 + index.md
  → 综合已有笔记回答
  → 引用来源笔记
  → 如果答案有价值，自动保存为新的query笔记
```

### 第三层：智能维护

#### 8. 健康检查（Lint）

```
🔍 知识库健康度: 87/100

⚠️ 发现3个问题：
1. "RAG架构"没有任何笔记链接到它（孤儿页面）
2. "GPT-4定价"最后更新于89天前（可能过时）
3. 两篇笔记对Transformer发明年份说法矛盾

[一键修复]  [逐个查看]
```

#### 9. 自动标签分类
- AI根据内容自动打标签
- 标签分类树自动生成和演化
- 用户可通过标签快速筛选笔记

#### 10. 知识图谱可视化
- 可视化笔记之间的链接关系
- 自动发现知识集群（Louvain社区检测）
- 标记知识空白（孤立的笔记群）
- 标记桥梁笔记（连接不同领域的关键笔记）

### 第四层：高级功能（后置）

#### 11. MCP Server（可选）
- 编辑器暴露为MCP Server
- Claude Code / Cursor等外部Agent可以操作笔记库
- 高级用户可命令行批量操作

#### 12. 多模型路由
- 实时结构化：DeepSeek / GPT-4o-mini（快、便宜）
- 复杂分析：Claude Sonnet / DeepSeek（高质量）
- 用户自带API Key

#### 13. 协作和同步
- 基于Git的版本控制
- 可选云同步
- 块级diff审核

#### 14. 模板系统
- 会议记录、读书笔记、项目文档等模板
- AI根据内容自动推荐模板

### 技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| 桌面框架 | Tauri 2 + React | 轻量(10MB vs Electron 100MB+)，Rust后端安全快速 |
| 编辑器内核 | TipTap (ProseMirror) | WYSIWYG编辑器，不是代码编辑器，适合普通人 |
| MD解析 | remark + unified | 成熟的MD AST工具链 |
| LLM调用 | Vercel AI SDK | 统一多provider接口，流式支持好 |
| 搜索 | SQLite FTS5 | 零依赖，嵌入式，Tauri自带SQLite |
| 图谱可视化 | Sigma.js / D3.js | 知识图谱网络图 |
| 存储 | 本地.md文件 + SQLite元数据 | Local-first，不锁数据库 |
| 同步 | Git (可选) | 版本控制+协作 |

## 用户画像

### 主要用户
- **知识工作者**：研究员、分析师、产品经理、咨询师
- **内容创作者**：博主、公众号作者、技术作者
- **学生**：研究生、博士生（论文写作+文献管理）

### 不面向
- 程序员（他们用VS Code + Claude Code）
- 需要团队协作的企业（他们用Notion/飞书）
