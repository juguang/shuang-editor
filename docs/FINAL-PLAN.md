# AI原生MD编辑器 — 最终方案文档

> **写了就是知识，不用排版，不用管理，越写越聪明。**

---

## 一、产品定位

让不懂markdown的普通人，用纯文本写作，AI自动加格式、建链接、归类，自动积累成知识库。

**类比：**
- Notion让普通人不用写代码就能做数据库
- 我们让普通人不用写markdown就能拥有知识库

**核心假设：** 知识管理的价值在"结构和连接"，但维护结构的成本太高，普通人坚持不了。AI把维护成本降到零。

## 二、竞品分析（为什么这个赛道是空白）

### 2.1 竞品矩阵

| 产品 | 自动格式化 | 自动链接 | 面向普通人 | 本地优先 | 缺什么 |
|------|:---:|:---:|:---:|:---:|------|
| **我们的产品** | ✅ | ✅ | ✅ | ✅ | — |
| Mem.ai | ⚠️ 部分 | ⚠️ 部分 | ✅ | ❌ | 云端+自有格式，不做MD格式化 |
| Notion AI | ❌ | ❌ | ✅ | ❌ | 手动`/`命令，不自动 |
| Reflect | ❌ | ✅ | ✅ | ❌ | AI只做后处理(选中→改写) |
| Craft | ❌ | ✅ | ✅ | ❌ | 手动建块 |
| Anytype | ❌ | ✅ | ✅ | ✅ | 完全没有AI |
| Obsidian | ❌ | 手动 | ❌ | ✅ | 必须学md语法 |
| Obsidian+3插件 | ✅ | ✅ | ❌ | ✅ | 要装Burnish+SmartLinker+AutoLinker |
| td(终端) | ✅ 完全一致 | ❌ | ❌ | ✅ | 开发者工具，命令行 |
| Workflowy | ❌ | ✅ | ✅ | ❌ | 大纲工具，非MD |

**结论：没有任何一款产品同时做到这四件事。**

### 2.2 最接近的三个竞品

**① Mem.ai（拿过数千万美元融资）**
- 理念最接近："dump brain, we organize"
- 但：云端、自有格式、不做markdown自动格式化
- **验证了需求存在**，但他们的解法不对

**② Obsidian + Burnish/Synapse/SmartLinker插件**
- 功能最接近：AI整理 + AI链接 + AI关联
- 但：需要装3-4个插件在Obsidian里，普通人搞不定
- **验证了功能可行**，但门槛太高

**③ td（终端工具，仅2⭐）**
- 技术完全一致：纯文本→Claude→结构化markdown
- 但：命令行工具，给开发者用
- **验证了技术路线**，但受众错位

### 2.3 为什么没人做？

| 原因 | 说明 |
|------|------|
| **技术刚成熟** | 实时流式LLM+低延迟+低成本，2024-2025年才可行 |
| **用户错位** | 会MD的人不需要自动格式化；需要的人不用MD编辑器 |
| **行业惯性** | 所有人都在"给编辑器加AI"，没人想"如果AI一开始就在，编辑器该长什么样" |
| **碎片化** | Obsidian插件各自解决一小块，没人整合成一个完整产品 |

## 三、完整产品蓝图

### 第一层：写作（核心）

#### 1. 零语法纯文本写作
- 打开编辑器，直接打字，没有任何markdown语法
- 没有工具栏，没有源码/预览切换
- 所见即所得，像Word/Notion一样自然
- 底层文件是标准markdown，用户永远看不到`#`、`**`、`[[]]`

#### 2. 三层自动结构化流水线

```
Layer 1 — 实时（打字时，流式）：
  → 句号后自动分段
  → 列表自动编号
  → 关键词自动加粗
  → 用轻量模型（DeepSeek/GPT-4o-mini）

Layer 2 — 段落级（回车后，debounce 2秒）：
  → 识别段落主题，加heading
  → 识别实体名称，自动加[[]]链接
  → 工具调用：search_notes()搜索已有笔记
  → 工具调用：web_search()补充外部知识（可选）

Layer 3 — 文档级（保存时，后台异步）：
  → 自动生成标题
  → 更新index.md（目录）
  → 更新log.md（操作日志）
  → 补充交叉引用
  → 检测矛盾
```

#### 3. 语音写作
- 说话 → AI直接生成结构化markdown
- 不是简单语音转文字，是实时结构化

### 第二层：知识库（自动积累）

#### 4. LLM Wiki三层结构（自动维护，用户无感）

```
notes/
├── index.md          ← AI自动更新的目录
├── log.md            ← AI自动追加的操作日志
├── raw/              ← 拖入的原始素材（不可变）
│   ├── articles/
│   ├── papers/
│   └── assets/
├── notes/            ← 用户的笔记（自动结构化后）
│   ├── concepts/     ← AI自动识别的概念页
│   ├── entities/     ← AI自动识别的实体页
│   └── daily/        ← 日志类笔记
└── SCHEMA.md         ← 自动生成的标签分类和规范
```

#### 5. 自动链接和交叉引用
用户写纯文本 → AI自动加`[[]]`链接 → 笔记之间形成网络

#### 6. 拖拽Ingest
拖入PDF/网页 → AI自动提取、摘要、建概念页、交叉引用

#### 7. 向知识库提问
"我之前写过关于XX的内容吗？" → AI检索+综合回答

### 第三层：智能维护

#### 8. 健康检查（Lint）
孤儿页面、断链、过时内容、矛盾检测

#### 9. 自动标签分类
AI根据内容自动打标签，标签树自动演化

#### 10. 知识图谱可视化
笔记之间的链接网络，自动发现知识集群

### 第四层：高级（后置）

| 功能 | 说明 |
|------|------|
| MCP Server | 暴露给外部Agent(Claude Code等)操作 |
| 多模型路由 | 实时用便宜模型，分析用强模型 |
| 协作同步 | 基于Git的版本控制 |
| 模板系统 | 会议记录/读书笔记/项目文档模板 |

## 四、技术架构

```
┌──────────────────────────────────────────────────┐
│                  UI Layer (React)                 │
│  ┌──────────┐  ┌──────────┐  ┌────────────────┐  │
│  │ 编辑区    │  │ 笔记列表  │  │ 搜索框         │  │
│  │ (TipTap  │  │ +标签树   │  │ (FTS5高亮)    │  │
│  │ WYSIWYG)│  │          │  │               │  │
│  └──────────┘  └──────────┘  └────────────────┘  │
├──────────────────────────────────────────────────┤
│              结构化引擎 (Core)                     │
│  ┌───────────┐  ┌───────────┐  ┌──────────────┐  │
│  │ 实时格式化  │  │ 段落级链接  │  │ 文档级整理    │  │
│  │ (流式LLM)  │  │ (LLM+工具) │  │ (后台异步)   │  │
│  └───────────┘  └───────────┘  └──────────────┘  │
├──────────────────────────────────────────────────┤
│              工具层 (4个函数)                      │
│  search_notes()  web_search()                    │
│  read_note()     write_note()                    │
├──────────────────────────────────────────────────┤
│              Storage (Rust/Tauri)                │
│  本地.md文件 + SQLite FTS5 + index.md + log.md   │
├──────────────────────────────────────────────────┤
│              LLM Provider                        │
│  DeepSeek / GPT-4o-mini / Claude (BYO-key)      │
│  Ollama本地模型 (可选)                            │
└──────────────────────────────────────────────────┘
```

### 技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| 桌面框架 | **Tauri 2 + React** | 10MB vs Electron 100MB+，Rust后端 |
| 编辑器 | **TipTap 2** (ProseMirror) | WYSIWYG，适合普通人，不是代码编辑器 |
| LLM SDK | **Vercel AI SDK** | 统一多provider，流式好 |
| 搜索 | **SQLite FTS5** | 零依赖，Tauri自带，支持中文 |
| MD解析 | **remark + unified** | MD↔TipTap互转 |
| 存储 | **本地.md文件** | Local-first，便携 |
| 样式 | **Tailwind CSS 4** | 快速开发 |

### 不用的技术（明确排除）

| 技术 | 为什么不用 |
|------|----------|
| ❌ 向量库/embedding | 200篇以内index.md + FTS5够了 |
| ❌ Agent Loop/ReAct | 不需要多步推理，LLM+function calling够了 |
| ❌ MCP Server（MVP阶段） | 过度工程，后置 |
| ❌ Electron | 太重(100MB+)，Tauri够用 |
| ❌ CodeMirror | 那是给开发者看源码的，我们要WYSIWYG |

## 五、MVP方案（4-6周）

### 只做三件事

```
1. 写 — 纯文本输入，AI自动加格式
2. 搜 — 全文搜索已保存笔记
3. 链 — 保存时自动链接到已有笔记
```

### 明确不做

agent loop、向量库、知识图谱、MCP、健康检查、语音、协作、拖拽Ingest、自动标签

### 用户故事

**张三打开编辑器，直接打字：**
```
今天开了个会，讨论了新产品方案。王总提出了三个方向，
一是做AI编辑器，二是做数据平台，三是做咨询服务。
我觉得AI编辑器最有前途，因为市场缺口大。
```

**AI自动结构化（张三全程无感）：**
```markdown
## 会议记录

今天开了个会，讨论了新产品方案。**王总**提出了三个方向：

1. 做**AI编辑器**
2. 做**数据平台**
3. 做**咨询服务**

我觉得**AI编辑器**最有前途，因为市场缺口大。
```

**一周后搜索"王总 产品方向"→ 立刻找到。**

### 开发计划

| 阶段 | 时间 | 交付 |
|------|------|------|
| **Week 1-2** 基础编辑器 | Tauri+React搭建，TipTap集成，笔记CRUD，文件读写 | 能用的纯文本编辑器 |
| **Week 3-4** AI结构化 | DeepSeek接入，实时格式化引擎，流式渲染，设置页 | **核心卖点可用** |
| **Week 5-6** 搜索+链接 | FTS5搜索，自动[[]]链接，暗色模式，打包发布 | **可日常使用** |

### 验收标准

**一个不懂markdown的人，5分钟内：**
1. 打开编辑器
2. 写一段纯文本
3. 看到"哇，自动变成格式化的笔记了"
4. 搜索到之前的笔记
5. 说"这个好用"

## 六、商业策略

### 开源 vs 商业
- **核心编辑器**：开源（MIT）
- **LLM API**：用户自带Key（BYO-key），零成本运营
- **可选云同步**：付费订阅（后期）
- **目标社区**：Obsidian觉得太复杂的人、Notion想要本地化的人

### 竞争定位

```
             AI能力强
                ↑
     Cursor ●   │   ★ 我们
               │
  ─────────────┼────────────→ 易用性
     Notion AI │   ● Obsidian
               │   ● Mem.ai
               │
             AI能力弱
```

我们的位置：**比Obsidian更AI原生 + 比Notion更本地化 + 比Mem.ai更开放（标准MD文件）**

## 七、风险和应对

| 风险 | 概率 | 应对 |
|------|:----:|------|
| AI格式化质量不稳定 | 中 | 多测试prompt工程，提供"撤销格式化"按钮 |
| LLM延迟影响体验 | 中 | Layer1用轻量模型（<500ms），Layer2 debounce 2秒 |
| 用户觉得"AI在改我的文字" | 高 | **关键设计：AI只加格式不改内容**，用户可随时关闭 |
| DeepSeek API不稳定 | 低 | 支持多provider切换（OpenAI/Ollama备用） |
| 搜索效果差（FTS5中文） | 中 | unicode61分词 + 后期可加jieba |
